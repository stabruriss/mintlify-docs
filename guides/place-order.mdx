This guide shows the typical flow for creating an order from a clinical workflow: a provider has selected a panel for a patient and your application now needs to materialize that decision in Vibrant's system.

## Prerequisites

- A sandbox or live API key with `orders_write` scope.
- A `provider_id` for the ordering clinician (provided by Vibrant during onboarding).
- A `test_code` from the [catalog](/concepts/test-catalog).

## Minimal request

The smallest valid order body:

```json
{
  "patient": {
    "first_name": "Avery",
    "last_name": "Nguyen",
    "date_of_birth": "1989-03-14",
    "sex": "female",
    "shipping_address": {
      "line1": "1200 Market St",
      "city": "San Francisco",
      "state": "CA",
      "postal_code": "94103",
      "country": "US"
    }
  },
  "provider_id": "prv_01HZ7M3K2N8R9Q",
  "tests": [{ "test_code": "GUT-ZOOMER-3.0" }],
  "sample_collection": { "method": "home_kit" }
}
```

## Full request, annotated

<CodeGroup>
  ```bash cURL
  curl -X POST "$VW_API_BASE/v1/orders" \
    -H "Authorization: Bearer $VW_API_KEY" \
    -H "Content-Type: application/json" \
    -H "Idempotency-Key: visit-4471-create" \
    -d @order.json
  ```

  ```ts TypeScript
  import { randomUUID } from "node:crypto";

  const order = await fetch(`${VW_API_BASE}/v1/orders`, {
    method: "POST",
    headers: {
      Authorization: `Bearer ${VW_API_KEY}`,
      "Content-Type": "application/json",
      "Idempotency-Key": `visit-${visitId}-create`,
    },
    body: JSON.stringify({
      patient: {
        external_id: visit.patient.id,
        first_name: visit.patient.firstName,
        last_name: visit.patient.lastName,
        date_of_birth: visit.patient.dob,
        sex: visit.patient.sex,
        email: visit.patient.email,
        shipping_address: visit.patient.address,
      },
      provider_id: visit.provider.vibrantId,
      tests: visit.orderedPanels.map((p) => ({ test_code: p.code })),
      sample_collection: { method: "home_kit" },
      metadata: { clinic_visit_id: visit.id },
    }),
  }).then((r) => r.json());
  ```

  ```python Python
  import requests, uuid

  payload = {
      "patient": {
          "external_id": visit.patient.id,
          "first_name": visit.patient.first_name,
          "last_name": visit.patient.last_name,
          "date_of_birth": visit.patient.dob.isoformat(),
          "sex": visit.patient.sex,
          "email": visit.patient.email,
          "shipping_address": visit.patient.address,
      },
      "provider_id": visit.provider.vibrant_id,
      "tests": [{"test_code": code} for code in visit.ordered_panels],
      "sample_collection": {"method": "home_kit"},
      "metadata": {"clinic_visit_id": visit.id},
  }

  r = requests.post(
      f"{VW_API_BASE}/v1/orders",
      headers={
          "Authorization": f"Bearer {VW_API_KEY}",
          "Idempotency-Key": f"visit-{visit.id}-create",
      },
      json=payload,
  )
  r.raise_for_status()
  ```
</CodeGroup>

### Patient

<ResponseField name="external_id" type="string">
  Your system's stable identifier for the patient. Stored verbatim. Vibrant never uses it as a join key — it's purely for your reconciliation.
</ResponseField>

<ResponseField name="sex" type="enum" required>
  Drives the reference range used in the report. Pass the patient's biological sex when known. Use `unknown` when it isn't safe to assume.
</ResponseField>

<ResponseField name="shipping_address" type="object" required>
  Where the kit is delivered. Required even for `in_clinic_draw` orders so the lab can mail backup supplies if needed.
</ResponseField>

### Tests

`tests` is an array — orders can bundle multiple panels run on the same sample (or shipped together as a multi-tube kit). The lab will produce one report per panel.

```json
{
  "tests": [
    { "test_code": "GUT-ZOOMER-3.0" },
    { "test_code": "MICRONUTRIENTS", "addons": ["OMEGA-3-INDEX"] }
  ]
}
```

### Sample collection

| `method` | When to use |
| --- | --- |
| `home_kit` | Default. A kit ships to the patient's address; they collect at home and drop in mail. |
| `in_clinic_draw` | Patient is collecting in your clinic; you'll ship the sample with the supplied label. |
| `patient_service_center` | Patient visits a partnered draw site. Requires `service_center_id`. |

### Billing

| `payer` | Behavior |
| --- | --- |
| `account` (default) | Vibrant invoices your account at end of month. |
| `patient` | Patient is charged at order time using `payment_method_id`. Requires the payment-methods API (coming soon). |
| `insurance` | Vibrant submits to insurance; balance billed to patient. Requires onboarding. |

### Metadata

Up to 20 string key/value pairs. Use it to store your foreign keys (visit IDs, EHR encounter IDs) so the order is reconcilable even when patients reach out directly.

## Response

```json
{
  "id": "ord_01HZ7M3K2N8R9Q",
  "status": "created",
  "patient": { "id": "pat_01HZ7M3K2N8R9Q", "..." },
  "provider_id": "prv_01HZ7M3K2N8R9Q",
  "tests": [{ "test_code": "GUT-ZOOMER-3.0" }],
  "sample_collection": { "method": "home_kit", "kit_shipping_speed": "standard" },
  "billing": { "payer": "account", "amount_cents": 49900, "currency": "USD" },
  "created_at": "2026-05-04T14:22:11Z",
  "updated_at": "2026-05-04T14:22:11Z"
}
```

Persist `id` keyed by your `metadata.clinic_visit_id` (or whatever you passed). That's how you'll find the order later when status changes arrive on webhooks.

## Common errors

<AccordionGroup>
  <Accordion title="422 validation_error — test_not_available" icon="ban">
    The `test_code` isn't enabled for your account or isn't valid in the patient's state. Re-fetch the catalog or contact onboarding.
  </Accordion>
  <Accordion title="409 conflict — idempotency_key_reuse" icon="rotate">
    You sent the same `Idempotency-Key` with a different body. Either reuse the original body or pick a new key.
  </Accordion>
  <Accordion title="402 payment_required" icon="credit-card">
    The account has no active payment method. Add one in **Settings → Billing** or have admin enable invoiced billing.
  </Accordion>
</AccordionGroup>
