Today, every order embeds the patient inline. That's fine for one-off integrations but creates duplicates when the same patient orders multiple times. The upcoming Patients API lets you persist a patient once and reference the ID on subsequent orders.

## Why this matters

## Planned endpoints

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/v1/patients` | Create a patient. |
| `GET` | `/v1/patients` | List/search patients. |
| `GET` | `/v1/patients/{patient_id}` | Retrieve. |
| `PATCH` | `/v1/patients/{patient_id}` | Update mutable fields. |
| `GET` | `/v1/patients/{patient_id}/orders` | List orders. |
| `GET` | `/v1/patients/{patient_id}/reports` | List reports. |

## Migration path

When the Patients API ships, existing inline patient blocks on orders will continue to work. Vibrant will retroactively create patient resources for every distinct `external_id` already on file, so your existing data shows up under `/v1/patients` automatically. New code can begin referencing `patient_id` on `POST /v1/orders`:

```jsonc
// Today
{ "patient": { "first_name": "...", "..." } }

// Future
{ "patient_id": "pat_01HZ7M3K2N8R9Q" }
```

Both forms will be accepted indefinitely. Migration is opt-in.
