The Payment Methods API will let you tokenize cards and bank accounts for both your account itself and for individual patients. Once available, the `billing.payer` field on orders becomes meaningful — you can split charges between the account, the patient, or insurance.

## Planned endpoints

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/v1/payment-methods` | Tokenize and attach a card or bank account. |
| `GET` | `/v1/payment-methods` | List methods on file. |
| `GET` | `/v1/payment-methods/{id}` | Retrieve. |
| `DELETE` | `/v1/payment-methods/{id}` | Detach. |
| `POST` | `/v1/payment-methods/{id}/set-default` | Make default for the owning patient or account. |
| `GET` | `/v1/orders/{order_id}/charges` | List charges. |
| `POST` | `/v1/charges/{charge_id}/refund` | Refund a charge. |

## Tokenization model

Vibrant will integrate with Stripe Elements / Stripe.js for tokenization. Your frontend never sends raw card numbers to Vibrant; you collect a Stripe `payment_method` token and POST it to `/v1/payment-methods`:

```json
{
  "owner": { "type": "patient", "patient_id": "pat_01HZ7M3K2N8R9Q" },
  "stripe_payment_method": "pm_1NXyZ...",
  "set_as_default": true
}
```

This keeps you out of PCI scope: only Stripe sees the PAN, and Vibrant only stores the token.

## Order integration

Once available, orders will accept a `payment_method_id` directly:

```json
{
  "billing": {
    "payer": "patient",
    "payment_method_id": "pm_01HZ7M3K2N8R9Q"
  }
}
```

The card is authorized at order creation and captured when the lab marks the order `sample_received`. If the order is `cancelled` or `rejected`, the authorization is voided.
