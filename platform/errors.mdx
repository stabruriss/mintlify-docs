The Portal API uses standard HTTP status codes alongside a JSON envelope that gives you a stable, machine-friendly handle for each failure mode.

## Envelope

```json
{
  "error": {
    "type": "validation_error",
    "code": "test_not_available",
    "message": "Test code GUT-ZOOMER-3.0 is not enabled for your account.",
    "param": "/tests/0/test_code",
    "request_id": "req_01HZ8R2W6F4P8Q"
  }
}
```

| Field | Description |
| --- | --- |
| `type` | High-level category. Use this for branching logic. |
| `code` | Stable, narrow identifier. Use this for precise messaging. |
| `message` | Human-readable description. May change over time — don't match on it. |
| `param` | JSON pointer to the offending field, when applicable. |
| `request_id` | Always present. Include it in support requests. |

## Status codes

| Status | `type` | When |
| --- | --- | --- |
| `400` | `invalid_request` | Body could not be parsed as JSON. |
| `401` | `authentication_error` | Missing, malformed, or revoked API key. |
| `403` | `permission_error` | Key lacks the required scope. |
| `402` | `payment_required` | No active payment method on the account. |
| `404` | `not_found` | Resource does not exist or is not visible to your account. |
| `409` | `conflict` | Resource is in a state that does not allow this action (e.g. cancelling a received order). |
| `422` | `validation_error` | One or more fields failed validation. `param` points to the first issue. |
| `429` | `rate_limited` | Too many requests. See [Rate limits](/platform/rate-limits). |
| `5xx` | `server_error` | Vibrant outage. Retry with backoff. |

  </Accordion>

## Retry policy

| `type` | Retry? | How |
| --- | --- | --- |
| `invalid_request`, `permission_error`, `validation_error` | No | Fix the request. |
| `authentication_error` | No | Replace the key. |
| `not_found`, `conflict` | No | Inspect resource state and adjust. |
| `payment_required` | No | Resolve billing first. |
| `rate_limited` | Yes | Honor `Retry-After`. |
| `server_error` | Yes | Exponential backoff up to 60s. Use `Idempotency-Key` so retries are safe. |

## Reference example

```ts
async function placeOrder(payload, attempt = 0): Promise<Order> {
  const res = await fetch(`${VW_API_BASE}/v1/orders`, {
    method: "POST",
    headers: {
      Authorization: `Bearer ${VW_API_KEY}`,
      "Content-Type": "application/json",
      "Idempotency-Key": payload.idempotency_key,
    },
    body: JSON.stringify(payload),
  });

  if (res.ok) return res.json();

  const { error } = await res.json();

  if (error.type === "rate_limited" && attempt < 5) {
    const wait = Number(res.headers.get("Retry-After") ?? 1) * 1000;
    await sleep(wait);
    return placeOrder(payload, attempt + 1);
  }

  if (error.type === "server_error" && attempt < 3) {
    await sleep(2 ** attempt * 1000);
    return placeOrder(payload, attempt + 1);
  }

  throw new VibrantError(error);
}
```
